# Contributing Guidelines

Thank you for your interest in contributing to the Simple Interest Calculator project. Contributions are welcome and appreciated. This document provides guidelines for contributing to the project.

---

## How to Contribute

You can contribute in several ways:

### Reporting Bugs

If you find a bug, please create an issue that includes:

* A clear and descriptive title.
* Steps to reproduce the problem.
* Expected behavior.
* Actual behavior.
* Screenshots (if applicable).
* Browser and operating system information.

---

### Fixing Bugs

Bug fixes are always welcome.

To submit a bug fix:

1. Fork the repository.
2. Create a new branch for your fix.
3. Make the necessary changes.
4. Test your changes thoroughly.
5. Commit your changes with a clear commit message.
6. Submit a pull request.

Example:

```text
git checkout -b fix-input-validation
```

---

### Documentation Improvements

Documentation contributions are encouraged.

Examples include:

* Correcting spelling or grammar mistakes.
* Improving project explanations.
* Adding examples and screenshots.
* Enhancing installation instructions.
* Updating README files.

---

### Feature Enhancements

Suggestions for new features are welcome.

Possible enhancements:

* Compound Interest Calculator.
* Currency formatting.
* Input validation and error messages.
* Mobile responsive design.
* Dark mode support.
* Interest comparison tools.

Before implementing a major feature, please open an issue to discuss the proposed changes.

---

## Development Process

1. Fork the repository.
2. Clone your fork locally.

```text
git clone https://github.com/your-username/simple-interest-calculator.git
```

3. Create a feature branch.

```text
git checkout -b feature-name
```

4. Make your changes.

5. Commit your changes.

```text
git commit -m "Add feature description"
```

6. Push to your fork.

```text
git push origin feature-name
```

7. Open a Pull Request.

---

## Pull Request Guidelines

Please ensure that:

* Code is clean and readable.
* Existing functionality is not broken.
* Changes are properly tested.
* Documentation is updated when necessary.
* Commit messages are meaningful.

---

## Code Style

* Use consistent HTML formatting.
* Follow standard CSS conventions.
* Write clear and maintainable JavaScript code.
* Use meaningful variable and function names.
* Keep the project simple and beginner-friendly.

---

## Questions and Support

If you have questions, suggestions, or ideas for improvement, feel free to open an issue and start a discussion.

---

## Thank You

Your contributions help improve this project and make it more useful for everyone. We appreciate your time and effort in helping the project grow.
